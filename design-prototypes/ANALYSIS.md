# Open-source hub prototype analysis

## Why this surface

The current product already has strong installation and operations mechanics.
The largest public-facing UX gap is not another feature list; it is helping a
self-hosting adopter answer four questions before installation:

1. Who maintains this project?
2. What rights and third-party obligations apply?
3. Can I recover or roll back the installation?
4. Where do I report a security or support issue?

## Intended person and task

- **Person**: a developer, small-team operator, or open-source evaluator who
  arrived from GitHub and is considering a first self-hosted deployment.
- **Immediate task**: inspect trust evidence, understand the supported boundary,
  and copy the install command.
- **Next task**: complete the Docker guide and reach the first successful model
  connection without losing the path back to operations documentation.

## Design direction

- **Domain**: release ledger, checksum, provenance chain, local stack,
  maintainer signature, bilingual manual, recovery point.
- **Color world**: amber wax seal, cream paper, graphite terminal, muted brass,
  moss-green verification, clay-red warning.
- **Signature**: a connected trust rail beside a dimensional release ledger.
- **Rejected defaults**: centered SaaS hero, three equal feature cards,
  purple/blue AI glow, testimonial or logo wall, generic privacy superlatives.

## What the prototype tests

- Whether legal, provenance, operations, and security documents can form the
  primary narrative rather than footer-only links.
- Whether Korean and English can switch in place without duplicating layouts.
- Whether a real maintainer signature creates trust without turning into a
  testimonial.
- Whether the install command remains prominent after moving trust evidence
  ahead of generic marketing claims.

## Candidate production work

1. Build a public `/open-source` or `/about` route from these primitives.
2. Replace prototype-relative Markdown links with rendered documentation routes.
3. Drive release facts from a checked manifest instead of static copy.
4. Add a real documentation parity status generated by CI.
5. Expose SBOM, checksum, tag, and supported-version evidence after those gates
   are implemented.
6. Reuse the maintainer signature and policy links in the repository homepage,
   installation wizard, and About dialog without duplicating policy text.

## Visual QA evidence

Checked in a real Chromium browser on 2026-07-12:

- 375 × 812: no page-level horizontal overflow; Korean line breaking passed.
- 768 × 1024: tablet layout and trust rail passed.
- 1280 × 900 and 1440 × 1000: hero hierarchy and release ledger passed.
- KO → EN switched the document language and hero copy correctly.
- Copy control returned visible `COPIED` feedback.
- All eight same-origin policy/document links returned HTTP 200.
- Final console result: 0 errors, 0 warnings.

The artifact is a standalone direction prototype, not a shipped product claim.
